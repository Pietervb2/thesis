\section{Model setup}
\label{sec::modelsetup}

% \todo[inline]{dit nalezen en aanpassen met eigen aantekeningen}
The simulation model is implemented in Python and organized into three folders. The \texttt{baseclasses} folder contains the classes representing the network, the simulation object, and the physical components. The \texttt{constants} folder contains JSON files that define the physical parameters of each component (pipe dimensions, heat exchanger constants, pump curve coefficients, and thermostatic bypass valve settings), allowing parameters to be easily changed. The \texttt{figures} folder contains the simulation output, consisting of generated plots and corresponding data in \texttt{.csv} format. Next to these folders, two scripts are the entry points of the model: \texttt{test.py}, which builds and runs simulations, and \texttt{BayesianOptimization.py}, which wraps the simulation in the constrained Bayesian Optimization approach of Chapter~\ref{chapter:ControlPolicy}.

The physical components listed in Eneco's model are implemented as class objects, with a general valve class that combines the valves in the heat exchangers and the thermostatic bypass valve. For the consumer, network, and simulation, class objects are also introduced. Where the simulation object is specifically created for plotting and data storage. 

The function \texttt{normal\_run()} in \texttt{test.py} can be used to run standalone simulations and to perform the benchmark simulation. This function allows a user-defined supply temperature $T_{s}$, either constant or time-varying. In contrast with the function \texttt{optimization\_run()}, which instead internally constructs the supply temperature following the lumped function definition of the supply temperature used for the Bayesian Optimization approach presented in Section ~\ref{sec::controlInputs}. The constants of the lumped function can be separately defined to run the simulation, but the function is also used in \texttt{BayesianOptimization.py}. This file creates a cost function object class and calls the \texttt{optimization\_run()} to obtain function evaluations for the Bayesian Optimization approach. The control policy is executed by running \texttt{run\_bo.py}. 

Both \texttt{\_run} functions first create a network by running on \texttt{network\_builder()}. It creates a \texttt{Network} object, builds the graph structure, and fills it with the relevant components via the \texttt{add\_*} functions. The relevant physical parameter values of the components are loaded through \texttt{read\_*\_data()}. 

By implementing the theory of Chapter~\ref{chapter:sysmodel} in combination with extensive logging of the system data, the inlet and outlet temperatures of the pipes, the mass flow rates, and the relevant internal variables of the heat exchangers and valves, the model provides insight into the dynamics of the Decentralized Solution, satisfying the first modeling requirement. The modular 
structure of the simulation model leaves room to extend the network with new components or alternative component designs. Together with the use of the Python programming language, this makes the model directly applicable for Eneco's engineers and provides a framework for further development. Hereby confirming to model requirements two and four. The model supports the Bayesian Optimization control strategy in Chapter~\ref{chapter:ControlPolicy} and implements P-control for the thermostatic bypass valve and PI-control for the valves within the heat exchanger. The extensive data logging additionally provides the foundation for testing alternative control strategies, although these would still need to be implemented within the model, satisfying model requirement three. 

% which assembles the 23-floor network programmatically. It first places the supply pump and the heat exchanger of the first floor, then iterates over the remaining consumers, adding per floor the four junction nodes, the supply and return pipes, and the heat exchanger with its consumer. Finally, the thermostatic bypass valve is connected between the supply and return side at the top floor. Because the network is generated in a loop, the number of floors, the pipe types per segment, and the consumer profiles can be changed freely, which keeps the model scalable to other DCS configurations.

\subsection{Kirchhoff's hydraulic laws}
% \todo[inline]{ingaan op hoe het netwerk van non-linear equations eruit ziet dus met matrix en shit}
The Kirchhoff laws yield a system of nonlinear equations, which are set equal to zero to obtain the mass flow rates throughout the system at timestep $k$.

\begin{equation}\label{eq::KirchoffNR}
F(\dot{\mathbf{m}}(k)) = 
\begin{bmatrix}
    A_{\text{red}}\, \dot{\mathbf{m}}(k) \\[6pt]
    L \, \mathbf{\Delta p_e} (\dot{\mathbf{m}}(k))
\end{bmatrix}
= \mathbf{0},
\end{equation}
where $F(\dot{\mathbf{m}}(k)) \in \mathbb{R}^{(|\mathcal{L}|+|\mathcal{N}|-1)}$ is composed of two blocks: the nodal mass balance residuals $A_{\text{red}}\,\dot{\mathbf{m}}(k) \in \mathbb{R}^{|\mathcal{N}|-1}$, one equation per node excluding the reference node, and the pressure drop residuals $L \, \mathbf{\Delta p_e} (\dot{\mathbf{m}}(k)) \in \mathbb{R}^{|\mathcal{L}|}$, one equation per fundamental loop, with the vector $\dot{\mathbf{m}}(k) \in \mathbb{R}^{|\mathcal{E}|}$ containing all the mass flow rates of the edges at timestep $k$ and $\mathbf{\Delta p_e} (\dot{\mathbf{m}}(k)) \in \mathbb{R}^{|\mathcal{N}|}$ as the vector containing the pressure drop over all the edges. 

The Hardy-Cross \cite{HardyCross} and Newton-Raphson \cite{NewtonenHard} methods are often used to solve such systems \cite{KUNTUAROVA}. These iterative methods start with an initial guess of the flow and adjust it iteratively until it converges. While the Hardy-Cross method iterates through each loop independently, the Newton–Raphson method iterates through all loops simultaneously \cite{NewtonenHard}. As a result, Newton-Raphson generally converges more quickly than Hardy-Cross. While there are several articles claiming faster convergence or more accurate determination of the mass flow rates in cases with low flow rates \cite{STEVANOVIC, ARSENE}, Newton-Raphson was selected for this work, given the use of Newton-Raphson type solvers by popular simulation tools such as Pandapipes \cite{pandapipes} and OpenModelica \cite{OpenModelica} and its widely available documentation. 

% The developed simulation model uses the root function of scipy to \eqref{eq::KirchoffNR}, using the default method apply the modified Powell method. 

% - resolution of the hydraulic dynamics is set to 4 number after the komma
% - Over the course of a whole simulation the solver sometimes tends to not solve. In the rare moment this happesn it stays below 20 occurances, on a full day simulation with a timestep of 1 second. Then $\min F(x)$ is performed where the mass flow rates of the previous timesteps are used to warm start the minimization.  

The developed simulation model solves \eqref{eq::KirchoffNR} using the \texttt{root} function of the \texttt{SciPy} \cite{scipy} package, with its default method, the modified Powell method. The Jacobian is derived analytically to aid the solver. The hydraulic dynamics are resolved to five decimal places. Over the course of a full simulation, the solver occasionally fails to converge. These occurrences are rare: fewer than 20 instances over a full-day simulation with a time step of one second, and in most simulations they do not occur at all. When this happens, the problem is solved by minimizing $\min ||F(x)||^{2}$ using the L-BFGS-B algorithm, with the mass flow rates from the previous time step used to warm-start the minimization.

$\dot{Q}_h$, from that at the secondary ('cold') side, $\dot{Q}_c$, both defined in ~\eqref{eq::hotandcolheatratetransfer}.

\subsection{Thermodynamic pipeline PDE}
To validate the implementation of the Node Method, the predicted outlet 
temperature is compared with experimental measurements from van der Heijde et 
al.\ \cite{VANDERHEIJDE2017158}, who used these measurements to validate their 
own thermo-hydraulic pipeline model. The test setup consists of a single steel 
pipe of 39 m, of which the parameters were already presented in 
Table~\ref{tab:pipe_heijden_geometry}. The inflowing water is heated by a 
boiler and the volume flow rate is measured, where the mass flow rate is kept 
constant during each experiment. Before each experiment, the pipe was flushed 
with water from the municipal water supply for approximately ten minutes to 
bring the system to steady state, after which the boiler was switched on and 
the water was heated to the desired inlet temperature. The conditions of the 
four experiments are summarized in Table~\ref{tab::heijden_conditions}.

Besides the experimental data, the Node Method is compared with a commonly 
used pipe model: the dynamic pipe model 
(\texttt{Modelica.Fluid.Pipes.DynamicPipe}) from the Modelica Standard Library 
(MSL). This model solves the pipeline dynamics using a Finite Volume Method, 
for which the pipe was discretized into 70 control volumes and by default solved with an implicit solver. For each experiment, the measured inlet temperature is applied as input to 
both models. The resulting outlet temperatures are compared with the 
measurements, as shown in Figure~\ref{fig::pipe_validation}. The RMSE of both 
models with respect to the experimental data is presented in 
Table~\ref{tab::pipe_validation_rmse}. The Node Method outperforms the Finite Volume model in all four experiments. Based on these results, the 
Node Method implementation is considered validated for the pipeline thermodynamics.
